# Money convention ‚Äî TAQINOR OS

## Mono-currency (MAD) invariant ‚Äî YDATA22

TAQINOR OS is **mono-devise MAD** today: every monetary amount persisted anywhere
in the ERP is understood to be Moroccan Dirhams unless a model explicitly says
otherwise. The invariant, spelled out so it can be checked and never assumed
silently:

> **"montant + devise voyagent ensemble."** Any monetary amount either (a)
> lives on a model whose ONLY currency is MAD by convention (no `devise` field
> needed, no ambiguity possible today), or (b) if the model can legitimately
> hold a non-MAD amount (multi-currency purchasing, quoting in a foreign
> currency, FX revaluation), it MUST carry an explicit `devise`/`devise_defaut`
> field next to the amount ‚Äî never a bare number whose unit depends on
> unwritten context.

Two families already do (b) correctly and are the reference pattern:

- `apps.ventes.Devis` / `apps.ventes.Facture` ‚Äî carry `devise` (default `MAD`)
  **and** `taux_change` (exchange rate to MAD) next to every monetary field.
- `apps.compta` multi-currency subsystem ‚Äî `TauxDevise` (rate table),
  `ItemOuvertDevise`, `EcartChange`, `ReevaluationCloture`/`LigneReevaluation`
  (FX revaluation) ‚Äî a dedicated, already-currency-aware layer.

Everything else in the ERP (the large majority of money-bearing models ‚Äî
`Facture`'s lines, `Paiement`, `BulletinPaie`, `Contrat`, fleet/RH/stock costs,
etc.) does (a): it stores a bare `DecimalField` amount with **no** `devise`
field, on the explicit assumption that the whole system is single-currency
MAD. `docs/currency-audit.md` (generated by `scripts/check_money_fields.py
--write`) lists every money-bearing model found today and which side of (a)/(b)
it falls on, so the mono-devise MAD hypothesis is a checked fact, not folklore.

**Why this matters going forward:** if TAQINOR OS ever grows a second currency
(supplier invoices in EUR, a foreign-currency loan, cross-border pricing), any
code that sums/compares two `DecimalField` amounts from two different models
must NOT silently assume both are MAD. The audit above is exactly the map of
"which amounts still need a `devise` field added" the day that happens ‚Äî this
is a **decision + inventory task (YDATA22)**, not a migration: no model is
changed here.
