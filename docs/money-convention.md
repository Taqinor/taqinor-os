# Money convention — TAQINOR OS

## Mono-currency (MAD) invariant — YDATA22

TAQINOR OS is **mono-devise MAD** today: every monetary amount persisted anywhere
in the ERP is understood to be Moroccan Dirhams unless a model explicitly says
otherwise. The invariant, spelled out so it can be checked and never assumed
silently:

> **"montant + devise voyagent ensemble."** Any monetary amount either (a)
> lives on a model whose ONLY currency is MAD by convention (no `devise` field
> needed, no ambiguity possible today), or (b) if the model can legitimately
> hold a non-MAD amount (multi-currency purchasing, quoting in a foreign
> currency, FX revaluation), it MUST carry an explicit `devise`/`devise_defaut`
> field next to the amount — never a bare number whose unit depends on
> unwritten context.

Two families already do (b) correctly and are the reference pattern:

- `apps.ventes.Devis` / `apps.ventes.Facture` — carry `devise` (default `MAD`)
  **and** `taux_change` (exchange rate to MAD) next to every monetary field.
- `apps.compta` multi-currency subsystem — `TauxDevise` (rate table),
  `ItemOuvertDevise`, `EcartChange`, `ReevaluationCloture`/`LigneReevaluation`
  (FX revaluation) — a dedicated, already-currency-aware layer.

Everything else in the ERP (the large majority of money-bearing models —
`Facture`'s lines, `Paiement`, `BulletinPaie`, `Contrat`, fleet/RH/stock costs,
etc.) does (a): it stores a bare `DecimalField` amount with **no** `devise`
field, on the explicit assumption that the whole system is single-currency
MAD. `docs/currency-audit.md` (generated by `scripts/check_money_monodevise.py
--write`) lists every money-bearing model found today and which side of (a)/(b)
it falls on, so the mono-devise MAD hypothesis is a checked fact, not folklore.

**Why this matters going forward:** if TAQINOR OS ever grows a second currency
(supplier invoices in EUR, a foreign-currency loan, cross-border pricing), any
code that sums/compares two `DecimalField` amounts from two different models
must NOT silently assume both are MAD. The audit above is exactly the map of
"which amounts still need a `devise` field added" the day that happens — this
is a **decision + inventory task (YDATA22)**, not a migration: no model is
changed here.

## Convention d'arrondi monétaire (YDATA8)

**Un seul point d'arrondi pour toute valeur en dirhams (MAD) :
`core.money.quantize_mad(value)`.**

### Pourquoi

- `round()` sur un `float` hérite de l'imprécision binaire IEEE-754
  (`round(2.675, 2)` vaut `2.67` en Python, pas `2.68`) et applique le
  "banker's rounding" (arrondi au pair le plus proche) — pas la politique
  "moitié vers le haut" attendue en comptabilité marocaine.
- `Decimal.quantize(..., rounding=ROUND_HALF_UP)` est déterministe et exact :
  `quantize_mad(Decimal('1.005')) == Decimal('1.01')`.

### Règle

- Toute valeur MONÉTAIRE finale (prix, montant, total HT/TTC, TVA, remise,
  acompte, solde…) affichée, stockée, ou comparée doit passer par
  `quantize_mad()` avant d'être figée — jamais un `round(x, 2)` brut.
- Les valeurs TECHNIQUES (kWc, surface, watt, débit, HMT…) restent hors
  périmètre : `round()` reste approprié, ce ne sont pas des montants.
- `scripts/check_money_rounding.py` est un garde ADVISORY (v1) : il signale
  tout `round()` appliqué à une valeur d'apparence monétaire dans les
  modules de pricing/tax (`apps/ventes/services.py`,
  `apps/ventes/quote_engine/builder.py`, `apps/compta/services.py`) et
  recommande `quantize_mad`. Il ne réécrit RIEN — les correctifs unitaires
  du code existant sont matière `ERROR_PLAN`, pas de ce garde. Un `round()`
  argent PRÉ-EXISTANT est baseline-allowlisté (`scripts/check_money_rounding.py`
  génère la liste à l'initialisation) ; seul un NOUVEAU site échoue la CI.

### Exemple

```python
from decimal import Decimal
from core.money import quantize_mad

total_ttc = quantize_mad(total_ht * (1 + taux_tva))
```
